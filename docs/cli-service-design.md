# TSMC CLI and Linux Service Design

## Summary

This document proposes a user-first CLI for self-hosting the TSMC backend on a Linux machine with a single, simple workflow:

```bash
uv tool install tsmc-server
tsmc service install --start
```

That workflow should:

- install the backend as an isolated tool with `uv`
- create a stable config and data layout under the user's home directory
- register a `systemd --user` service
- start the backend in the background
- leave the generated Markdown files in a predictable location that other local tools, like OpenClaw, can read directly

The CLI is primarily for self-hosted Linux use. Railway or other cloud deployments remain supported through normal ASGI/container deployment, but they are not the primary target for this UX.

## Why This Matters

From the user's perspective, the current backend is still a developer workflow:

- clone repo
- `cd backend`
- `uv sync`
- `uv run dev`

That is fine for contributors. It is not the right UX for a user who just wants the TSMC backend running in the background on their machine.

The self-hosted user wants four things:

1. A single install command.
2. A single command to make the service start and stay running.
3. A stable data directory they can point other tools at.
4. Clear status, logs, and upgrade commands when something goes wrong.

## User Experience Goals

- Zero repo checkout required for normal users.
- No Python virtualenv management by hand.
- No manual `systemd` unit editing for the common case.
- No hidden storage paths.
- No raw SQLAlchemy URLs in the common flow.
- No ambiguity about where Markdown exports live.
- Safe defaults for local-only use.

## Non-Goals

- Replacing the extension dev workflow.
- Solving multi-node or replicated deployments.
- First-class Windows or macOS background-service support in v1.
- First-class Railway management through the CLI.
- Full secret-management UX beyond a local env file in v1.

## Installation Constraints

The intended install mechanism is `uv tool install`, which installs a package into an isolated tool environment and exposes its console scripts on the user's `PATH`. `uv` also supports upgrading and uninstalling tool environments with `uv tool upgrade` and `uv tool uninstall`.

This proposal assumes:

- a PyPI-published package
- a `console_scripts` entry point named `tsmc`
- a Linux machine with `systemd --user`

Important package naming note:

- `tsmc` is already taken on PyPI.
- The recommended distribution name is `tsmc-server`.
- The installed command should still be `tsmc`.

That gives the user this install flow:

```bash
uv tool install tsmc-server
```

## Recommended Command Surface

### Canonical first-run flow

```bash
uv tool install tsmc-server
tsmc service install --start
```

This should be the documented happy path everywhere.

### Top-level commands

- `tsmc service install`
- `tsmc service uninstall`
- `tsmc service start`
- `tsmc service stop`
- `tsmc service restart`
- `tsmc service status`
- `tsmc service logs`
- `tsmc run`
- `tsmc doctor`
- `tsmc config show`
- `tsmc config path`
- `tsmc version`

### Commands we should not include in v1

- `tsmc setup`
- `tsmc cloud`
- `tsmc railway`
- `tsmc secrets set`
- `tsmc openclaw`

Those can be added later if the need is real. They add cognitive load without helping the first-run user.

## Command Behavior

### `tsmc service install`

Purpose:

- bootstrap the local config if it does not exist
- create data directories
- generate a user-scoped `systemd` unit
- register the service
- optionally start it

Recommended flags:

- `--start`: start immediately after install
- `--enable`: enable service for login-time startup
- `--host`
- `--port`
- `--data-dir`
- `--markdown-dir`
- `--public-url`
- `--force`: replace existing unit/config values that were previously generated by TSMC

Recommended default:

```bash
tsmc service install --start
```

Behavior:

1. Resolve config and data directories using XDG defaults.
2. Create config and env files if missing.
3. Create the Markdown and SQLite directories.
4. Write a `systemd --user` unit file.
5. Run `systemctl --user daemon-reload`.
6. Run `systemctl --user enable tsmc.service` if `--enable` or `--start` is given.
7. Run `systemctl --user start tsmc.service` if `--start` is given.
8. Print:
   - service name
   - local URL
   - config path
   - Markdown path
   - database path
   - next useful commands

If the user has no `systemd --user` session available, the command should fail clearly and explain that Linux self-hosted service mode currently requires `systemd --user`.

### `tsmc service status`

Purpose:

- answer "is it up?"
- show whether the service is enabled
- show whether the HTTP health endpoint responds

Output should include:

- systemd state
- last known PID
- configured bind address
- healthcheck result from `/api/v1/health`
- config path
- Markdown path
- database path

### `tsmc service logs`

Purpose:

- give the user the exact logs they need without making them learn `journalctl`

Recommended behavior:

- default: last 100 lines
- `-f` or `--follow`: stream logs
- `--since <duration>`

Implementation target:

```bash
journalctl --user -u tsmc.service
```

### `tsmc service uninstall`

Purpose:

- remove the user service cleanly
- optionally keep user data

Recommended flags:

- `--keep-data` default `true`
- `--purge-data`

Default behavior should be conservative:

- stop the service
- disable the service
- remove the unit
- keep config and data unless the user explicitly asks to purge them

### `tsmc run`

Purpose:

- run the packaged backend in the foreground without systemd
- useful for debugging and container-style deployment

Recommended flags:

- `--host`
- `--port`
- `--config`
- `--data-dir`
- `--markdown-dir`

This command is also the command the generated `systemd` unit should execute.

### `tsmc doctor`

Purpose:

- tell the user what is broken in one place

Checks should include:

- config file exists and parses
- env file exists
- data and Markdown directories exist and are writable
- configured port is free or already owned by the TSMC service
- `systemctl --user` is available
- service unit exists
- HTTP healthcheck succeeds if service is running

### `tsmc config show`

Purpose:

- print the effective config without forcing the user to inspect systemd or env vars

Should show:

- host
- port
- data dir
- markdown dir
- public URL if configured
- LLM backend

Secrets should be redacted.

### `tsmc config path`

Purpose:

- answer "where are my files?"

Should print:

- main config path
- env file path
- systemd unit path
- data dir
- markdown dir
- sqlite db path

## User-Facing Defaults

### Service model

Use `systemd --user`, not a system-level unit, in v1.

Why:

- no sudo required for the common case
- safer for a local-only app
- lower risk of writing files as root that the user cannot later access
- better fit for a browser extension talking to a service owned by the same user

### Bind address

Default:

- `127.0.0.1`
- port `8000`

Why:

- matches the current extension default
- keeps the backend local by default
- reduces exposure while auth is still minimal

### Storage layout

Use XDG defaults:

- config: `~/.config/tsmc/config.toml`
- env file: `~/.config/tsmc/tsmc.env`
- data: `~/.local/share/tsmc/`
- markdown: `~/.local/share/tsmc/markdown/`
- sqlite db: `~/.local/share/tsmc/tsmc.db`
- unit: `~/.config/systemd/user/tsmc.service`

If `XDG_CONFIG_HOME` or `XDG_DATA_HOME` are set, honor them.

### Logs

Use journald for service logs. Do not invent a second logging system in v1.

## OpenClaw Integration

This is a primary design concern.

OpenClaw needs direct access to generated Markdown files. That strongly favors self-hosting TSMC on the same machine as OpenClaw.

The CLI should make the Markdown path explicit and configurable:

```bash
tsmc service install --start \
  --data-dir /srv/tsmc \
  --markdown-dir /srv/tsmc/markdown
```

That gives the user a stable local path they can mount or point OpenClaw at directly.

Recommended user guidance:

- keep Markdown in a normal local directory
- do not bury it inside a disposable temp path
- prefer one explicit shared folder for TSMC and OpenClaw interop

The CLI does not need a dedicated `openclaw` subcommand in v1. The core requirement is simply that the Markdown directory is stable, configurable, and easy to discover.

## Configuration Model

The current backend is driven mostly by environment variables. That is fine internally, but it is too raw for the primary user flow.

Recommended split:

- `config.toml`: non-secret user configuration
- `tsmc.env`: secrets and environment overrides

### Example `config.toml`

```toml
[server]
host = "127.0.0.1"
port = 8000

[storage]
data_dir = "/home/alice/.local/share/tsmc"
markdown_dir = "/home/alice/.local/share/tsmc/markdown"

[processing]
llm_backend = "auto"

[service]
name = "tsmc"
```

### Example `tsmc.env`

```bash
TSMC_OPENAI_API_KEY=
TSMC_GOOGLE_API_KEY=
TSMC_CORS_ORIGIN_REGEX=chrome-extension://[a-p]{32}
```

### Runtime translation

The CLI should translate the friendly config into the backend's existing environment model:

- `TSMC_DATABASE_URL`
- `TSMC_MARKDOWN_DIR`
- `TSMC_LLM_BACKEND`
- `TSMC_CORS_ORIGIN_REGEX`

That lets the backend app remain mostly unchanged while giving the user a cleaner UX.

## Proposed systemd Unit

Recommended generated unit:

```ini
[Unit]
Description=TSMC backend
After=network.target

[Service]
Type=simple
ExecStart=/home/alice/.local/bin/tsmc run --config /home/alice/.config/tsmc/config.toml
EnvironmentFile=/home/alice/.config/tsmc/tsmc.env
Restart=on-failure
RestartSec=2
WorkingDirectory=/home/alice/.local/share/tsmc

[Install]
WantedBy=default.target
```

Notes:

- `ExecStart` should use the absolute path of the installed `tsmc` script.
- Do not rely on the user's shell `PATH`.
- Do not embed secrets in the unit file.

## Boot Behavior and Linger

There is one user-facing trap with `systemd --user`: a user service normally starts when the user session starts.

That means:

- on a desktop or laptop, this is often good enough
- on a headless Linux server, the user may expect the service to start at boot without login

Recommended v1 behavior:

- install a normal user service
- detect whether `loginctl` is present and whether lingering is enabled
- if lingering is not enabled, print a clear note after install:

```text
TSMC will start when your user session starts.
To keep it running at boot without login, enable lingering for your user:
  sudo loginctl enable-linger <user>
```

Do not try to silently escalate privileges to configure lingering in v1.

## Packaging Plan

### Distribution name

Recommended PyPI package name:

- `tsmc-server`

Reason:

- `tsmc` is already taken on PyPI
- the user can still get a clean CLI command named `tsmc`

### Console script

Use:

```toml
[project.scripts]
tsmc = "app.cli:main"
```

### Build and publish

Keep publishing from the backend package directory.

Recommended release path:

```bash
cd backend
uv build
uv publish
```

Recommended CI later:

- GitHub Actions trusted publishing to PyPI
- build wheel and sdist
- run backend tests before publish

## CLI Implementation Shape

Recommended module layout:

- `app/cli.py`: CLI entrypoint
- `app/cli_config.py`: config path resolution and TOML parsing
- `app/cli_service.py`: systemd unit generation and systemctl wrappers
- `app/cli_doctor.py`: environment and health checks

Recommended framework:

- `Typer` is a good fit for nested subcommands and help output
- `argparse` is acceptable if we want zero extra runtime dependencies

Recommendation:

- use `Typer`

Why:

- the CLI will have multiple subcommands
- help output matters for non-developer users
- completion and type-driven option parsing are useful

## Railway and Other Cloud Hosts

This CLI proposal is intentionally optimized for a local Linux machine, not Railway.

That is the right tradeoff because:

- the current app writes Markdown files to local disk
- SQLite is still the default database
- OpenClaw needs direct filesystem access to the Markdown output

Railway can still host the backend as a regular ASGI service, but that should be treated as a separate deployment mode:

- `tsmc run --host 0.0.0.0 --port $PORT`
- mounted persistent volume if SQLite and Markdown must persist
- no local OpenClaw filesystem integration

Recommendation:

- make self-hosted Linux the primary supported path
- keep Railway support as secondary, documented separately

## Failure Modes and UX Requirements

### Port already in use

Behavior:

- fail fast
- show the conflicting port
- tell the user how to re-run with `--port`

### Broken config

Behavior:

- show the config path
- show the exact invalid field
- do not dump a raw Python traceback by default

### Service installed but unhealthy

Behavior:

- `tsmc service status` should say:
  - systemd is active
  - HTTP healthcheck failed
  - last few log lines are available via `tsmc service logs`

### Missing API keys

Behavior:

- service still runs
- processing remains on heuristic fallback
- status output should show `llm_backend=auto (heuristic active)` when relevant

## Rollout Plan

### Phase 1

- publish package to PyPI under `tsmc-server`
- add `tsmc` console script
- implement `tsmc run`
- implement `tsmc service install|start|stop|restart|status|logs|uninstall`
- implement XDG config/data layout
- document OpenClaw shared Markdown path setup

### Phase 2

- add `tsmc doctor`
- add `tsmc config show|path`
- improve error formatting
- add shell completion docs

### Phase 3

- add backup/export commands if needed
- add migration tooling if SQLite schema evolution becomes more complex
- consider a macOS service manager story

## Final Recommendation

The v1 UX should optimize for one thing:

```bash
uv tool install tsmc-server
tsmc service install --start
```

Everything in the design should support that path cleanly.

The local self-hosted Linux service should be the primary product shape because it aligns with the current architecture and with the OpenClaw requirement for direct Markdown access.
